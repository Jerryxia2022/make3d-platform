# Phase05-C Slicing Result Parsing Design

Date: 2026-07-14
Status: design approved; Phase05-C implementation completed separately in `reports/phase05-c-parser-implementation-final.md`

## Scope

Phase05-C is limited to TEST G-code sample analysis and parser/storage design. The approved implementation remains a pure parser module and does not add migrations or quote integration.

This design phase does not:
- create a database migration
- modify automatic quote logic
- modify order amounts
- modify payment state
- modify WeChat Pay
- modify upload limits
- deploy production automatic slicing
- use real customer files
- use paid order files
- write slicing results to production database
- automatically change order status

## 3. Parsing Priority

Priority 1:

```text
PrusaSlicer-generated G-code statistics comments near the tail.
```

Examples:

```text
; filament used [mm] = 2116.64
; filament used [cm3] = 5.09
; total filament used [g] = 0.00
; estimated printing time (normal mode) = 24m 56s
; estimated printing time (silent mode) = 25m 44s
```

Priority 2:

```text
PrusaSlicer standard metadata/config comments.
```

Examples:

```text
; filament_type = PLA
; printer_model = Bambu Lab P1S
; nozzle_diameter = 0.4
; layer_height = 0.2
```

Priority 3:

```text
stdout from the slicing process.
```

Use stdout only as diagnostic context, not as the primary source for core pricing values.

Forbidden inference sources:

```text
G-code file size
STL volume
filename
customer/order context
```

Do not infer print time or material weight from those sources.

## 5. Unit Specification

Store core numeric values as integers:

```text
print_time_seconds: integer seconds
silent_print_time_seconds: integer seconds
filament_length_mm: decimal display value only; store as filament_length_microns or filament_length_um if needed
filament_volume_cm3: decimal display value only; store as filament_volume_mm3
filament_weight_g: decimal display value only; store as filament_weight_mg
layer_count: integer count
max_layer_z_mm: decimal display value only; store as max_layer_z_microns
gcode_size_bytes: integer bytes
```

Recommended DB conversions:

```text
24m 56s -> 1496 seconds
25m 44s -> 1544 seconds
2116.64 mm -> 2116640 microns
5.09 cm3 -> 5090 mm3
0.00 g -> 0 mg
20 mm -> 20000 microns
284994 bytes -> 284994 bytes
```

Do not store money or prices as floating point values in parser output.

## 6. Parse Status

Recommended statuses:

```text
pending
parsing
parsed
partial
failed
```

Definitions:

```text
pending: parse job exists but has not started
parsing: worker/parser has claimed the job and is reading output
parsed: G-code is valid and all quote-required core fields are present
partial: G-code is valid but one or more non-critical fields are missing
failed: G-code is invalid, field format is malformed, or parser cannot recognize the file
```

Rules:

```text
missing fields must not be converted to 0
zero must mean an explicit parsed numeric zero from source data
null must mean missing or not applicable
warnings must explain partial data and derived values
```

Recommended indexes:

```sql
CREATE INDEX idx_slicing_jobs_file_id ON slicing_jobs(file_id);
CREATE INDEX idx_slicing_jobs_status_created ON slicing_jobs(status, created_at);
CREATE INDEX idx_slicing_jobs_worker_status ON slicing_jobs(worker_id, status);
CREATE INDEX idx_slicing_jobs_cache_key
  ON slicing_jobs(input_sha256, profile_sha256, slicer_package_version, slicer_banner_version);
```

Version strategy:

```text
Do not make `file_id` globally unique unless the product decides that one file may only have one slice result.
Recommended: allow multiple rows per file_id and use cache key uniqueness for same input/profile/slicer.
```

Optional idempotency/caching constraint:

```sql
CREATE UNIQUE INDEX idx_slicing_jobs_cache_unique
ON slicing_jobs(input_sha256, profile_sha256, slicer_package_version, slicer_banner_version)
WHERE parse_status IN ('parsed', 'partial');
```

This permits re-slicing after profile changes and avoids duplicate successful work for identical input/profile/slicer combinations.

## 8. Idempotency Design

Idempotency key:

```text
input_sha256 + profile_sha256 + slice_params_sha256 + slicer_name + slicer_package_version + parser_version
```

`profile_sha256` alone is not sufficient because CLI parameters can override profile values. The parser/workflow must persist a normalized slice parameter object:

```json
{
  "material": "PLA",
  "printer_model": "Bambu Lab P1S",
  "nozzle_diameter_microns": 400,
  "layer_height_microns": 200,
  "fill_density_percent": 50,
  "support_mode": "none",
  "brim_width_microns": 0
}
```

The field order must be stable, and `slice_params_sha256` must be calculated from the canonical JSON string.

Rules:

```text
same key and parsed output exists: reuse cached result
same file_id but different profile_sha256: allow new slicing job
same file_id but different slicer version: allow new slicing job
same job retried after worker failure: increment attempt_count and preserve prior diagnostics
```

Do not overwrite a successful parsed result with a failed retry unless a new attempt record is intentionally created.

## 9. Cache Design

Cache target:

```text
avoid repeated slicing for identical input and slicing profile
```

Cache key:

```text
input_sha256
profile_sha256
slice_params_sha256
slicer_package_version
parser_version
```

Cache invalidation:

```text
profile content changes -> profile_sha256 changes -> cache miss
PrusaSlicer package changes -> version changes -> cache miss
parser semantics change -> parser_version changes -> cache miss
input file changes -> input_sha256 changes -> cache miss
```

Cache must not cross materials/processes if those affect CLI args. Material, nozzle, layer height, infill, support, brim, and printer model must be represented in normalized `slice_params_json` and `slice_params_sha256`.

## 9A. Metric Sources And Quote Readiness

Parsing success and quote readiness are separate concepts.

`parse.status` means only whether the parser successfully read and recognized the G-code. `validation.quote_ready` means whether parsed metrics are acceptable for a later pricing design.

Standard validation object:

```json
{
  "metrics_status": "valid",
  "quote_ready": false,
  "invalid_fields": [],
  "warnings": []
}
```

Metric source object:

```json
{
  "print_time_source": "gcode_tail_stat",
  "filament_length_source": "gcode_tail_stat",
  "filament_volume_source": "gcode_tail_stat",
  "filament_weight_source": "gcode_tail_stat",
  "layer_count_source": "derived_layer_markers",
  "max_layer_z_source": "derived_z_markers"
}
```

Allowed source names:

```text
gcode_tail_stat
gcode_config
derived_layer_markers
derived_z_markers
stdout
missing
```

For the current TEST G-code:

```text
parse.status = parsed
validation.metrics_status = warning
validation.quote_ready = false
validation.invalid_fields = ["filament_weight_mg"]
validation.warnings includes: explicit source weight is zero while filament volume is nonzero
```

The parser must not derive filament weight from volume and density. That belongs to a later separately approved pricing/metrics policy.

## 10. Parser Module Design

Approved file:

```text
worker/prusaslicer-result-parser.mjs
```

Implementation approval was granted after this design review. The resulting module remains inside the approved pure-parser boundary.

Module boundary:

```text
pure parser module
input: G-code path plus optional context metadata
output: standard JSON
```

Must not:

```text
access database
access WeChat Pay
modify orders
modify quotes
write production state
```

Parsing source order:

```text
1. Tail statistics comments
2. Tail config/metadata comments
3. stdout diagnostics
```

Recommended parser patterns:

```text
; filament used [mm] = <decimal>
; filament used [cm3] = <decimal>
; total filament used [g] = <decimal>
; estimated printing time (normal mode) = <duration>
; estimated printing time (silent mode) = <duration>
; <config_key> = <value>
;LAYER_CHANGE
;Z:<decimal>
```

Duration parser:

```text
support h/m/s combinations
reject unknown units
do not treat unparsable duration as 0
```

## 11. Parser Safety Design

Do not read unbounded G-code into memory.

Recommended file access:

```text
stat file first
reject empty file
reject files above configured max size unless explicitly allowed
read tail window first, for example last 1 MiB
if required fields are missing, grow to 2 MiB, then 4 MiB
set hard max read window, for example 8 MiB
stream structural layer marker counting if needed
```

Path safety:

```text
resolve real path
require path under worker-controlled output directory
reject path traversal
never read /app/uploads directly from this parser
```

Content safety:

```text
reject HTML/error pages
limit line length for comments
truncate stored warnings/errors
do not log full G-code
do not log customer-sensitive filenames in shared logs unless masked
```

Suggested read windows:

```text
tail stats/config: last 1 MiB, expandable to 8 MiB max
layer count: streaming line scan with max file size guard
```

## 12. Test Design

At least these cases:

```text
1. Complete PrusaSlicer G-code with normal and silent time, length, volume, and weight fields
2. Missing weight
3. Missing print time
4. Field order changed
5. Windows CRLF line endings
6. Linux LF line endings
7. Decimal formats including 0, 0.00, 5.09, and integer-looking values
8. Large G-code using tail-window parsing
9. Empty file
10. HTML error file
11. Corrupted or truncated G-code
12. Duplicate fields, with deterministic last/first policy
13. Unknown PrusaSlicer version/banner
14. Abnormal units
15. Malicious very long comments
```

Additional Make3D-specific cases:

```text
explicit zero is not missing
missing field remains null
layer_count derived from markers is warning-tagged
max_layer_z derived from Z markers is warning-tagged
cache key changes when profile_sha256 changes
parser does not import database/payment/order modules
```

## 13. Boundary With Automatic Quote

This parser design must not directly calculate price.

Allowed:

```text
extract normalized slicing metrics
report parse status and warnings
provide typed values for later reviewed pricing design
```

Forbidden in Phase05-C:

```text
updating order totals
updating quote totals
creating payment requests
changing payment status
opening customer-facing automatic slicing
assuming missing material weight equals zero
using file size/STL volume/filename to infer print time or material use
```

Future automatic quote integration must be a separate approved phase.

## 14. Phase04 Status Semantics

Current Phase04 sync statuses:

```text
pending
locked
verified
```

Meaning:

```text
verified means the local file SHA-256 has been verified after sync.
```

Future slicing completion must use independent `slicing_jobs` statuses. Do not reuse file sync status to represent slicing state.

Do not rename or reinterpret `verified` as `local_synced` unless a separate design and review approves it.

## 15. Risks

Known risks:

```text
PrusaSlicer output fields may vary by version/profile/printer
`total filament used [g]` may be zero when filament density is zero in baseline profile
layer count may not be explicitly emitted
config block values are not necessarily final production settings
large G-code files can create memory pressure if parser reads entire files
duplicate fields need deterministic handling
```

Mitigation:

```text
version parser
store missing fields separately
use tail-window parsing plus guarded streaming
store parser warnings
separate parser metrics from pricing decisions
include profile and slicer hashes in cache key
```
